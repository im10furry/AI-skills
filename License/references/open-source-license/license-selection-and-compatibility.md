# 开源许可证选择与兼容性筛查

## 先做四个判断

1. 你想要“尽量少限制”，还是希望修改后也继续开源。
2. 你是否需要明确专利授权。
3. 你是否分发二进制，还是只做 SaaS。
4. 你是否已经依赖带 copyleft 义务的第三方代码。

## 常见选择倾向

- `MIT`
  - 极简宽松，适合希望复用成本低、文本最短的项目。
- `BSD-2-Clause` / `BSD-3-Clause`
  - 与 `MIT` 类似，常见于基础库或延续既有生态习惯的项目。
- `Apache-2.0`
  - 宽松，但比 `MIT` 多了明确的专利授权与专利终止相关条款；企业和基础设施项目常见。
- `MPL-2.0`
  - 文件级弱 copyleft。通常适合希望“改了核心文件就回流，但不强拉整仓库”的项目。
- `LGPL-3.0`
  - 适合库场景，允许在一定条件下与专有软件组合，但义务比 `MPL-2.0` 更复杂。
- `GPL-3.0`
  - 强 copyleft，分发衍生作品时通常要求相应开源义务。
- `AGPL-3.0`
  - 在 `GPL-3.0` 基础上进一步覆盖网络服务场景，适合希望 SaaS 侧修改也回流的项目。

## 初步兼容性筛查问题

- 你是“引用接口”还是“复制代码”。
- 你是静态链接、动态链接、插件加载，还是进程间通信。
- 你是否把第三方代码改进后一起分发。
- 你是否只在服务端内部运行而不分发。
- 你是否准备混合多个许可证源码文件。

## 输出建议模板

- 推荐候选：
  - `Apache-2.0`，因为项目要商业友好、要有专利授权、也不希望 copyleft。
- 兼容性前提：
  - 以下判断按“仓库分发源码 + 二进制发布”前提成立；如果只是内部部署或纯 SaaS，结论可能变化。
- 还需确认：
  - 是否已有 `GPL` / `AGPL` 依赖、是否允许双许可证、是否要保留商标限制。

## 谨慎点

- 不要在没看依赖树和分发方式前，武断说“完全兼容”。
- 不要把“公司不允许商用”的文本称为开源许可证。
- 不要忽略图片、字体、文档、模型、数据集这些非代码资产的单独许可证。
