# Copyleft、SaaS 与链接边界

## 为什么先分边界

- 很多“兼容不兼容”其实取决于你是复制代码、链接库、加载插件，还是只是调用网络接口。
- 这些判断适合做工程与合规初筛，但不应被表述成正式法律结论。

## 先分五类关系

1. 直接复制或改写第三方源码
   - 风险最高，通常最容易触发原许可证义务。
2. 静态链接
   - 对 copyleft 分析通常比纯 API 调用更敏感。
3. 动态链接
   - 仍可能触发义务，但实践争议通常比静态链接更多。
4. 插件、扩展、脚本接口、独立进程通信
   - 需要看耦合程度、分发方式和是否属于同一作品边界。
5. 纯网络调用或托管服务
   - 对 `GPL` 常常与“分发”问题相关；对 `AGPL` 则要额外关注网络服务提供场景。

## 输出兼容性结论时的前提模板

- 以下判断基于 `[源码复制 / 静态链接 / 动态链接 / 插件 / 纯 SaaS]` 场景。
- 如果实际关系不同，例如只是独立服务间 API 调用，结论可能改变。
- 如果项目会分发 SDK、客户端二进制或容器镜像，需要把分发形态单独列出来复核。

## 典型提醒

- 不要把 “只是内部使用” 和 “对外提供 SaaS” 混为一谈。
- 不要把 “没改对方代码” 自动等同于 “一定没有任何许可证义务”。
- 不要忽视容器镜像、桌面安装包、移动端 App 这些也是分发载体。

## 常见输出结构

- 初筛结论：
  - 在“服务端独立进程 + 仅 API 调用”前提下，风险通常低于直接复制源码。
- 仍需确认：
  - 是否存在共享源码、静态链接、打包分发、插件捆绑。
- 合规动作：
  - 复核依赖声明、构建产物、容器镜像、客户端 SDK 和第三方 notice。
