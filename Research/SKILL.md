# Research

用这个 skill 作为调研类任务的单入口。先判断用户要解决的是哪类决策问题，再选择市场调研、功能调研、新产品调研或现状调研的路径，不要一上来就堆资料。

## Quick Start

1. 先把研究问题写成一句决策句。
   - `我们是否应该进入这个市场？`
   - `这个功能值不值得做？`
   - `这个新产品是否有机会成立？`
   - `当前业务或系统到底卡在哪里？`
2. 如果用户只说“做个调研”，优先补 5 个关键信息：
   - 研究对象
   - 研究范围
   - 时间范围
   - 已有假设
   - 输出形式
3. 调研结果默认分成三层：
   - 事实
   - 推断
   - 建议
4. 涉及市场规模、竞品动态、价格、政策、平台规则、用户行为等强时效信息时，优先联网验证最近信息。

## Workflow

1. 识别研究类型。
   - 市场调研：判断市场空间、竞争格局、渠道和价格带。
   - 功能调研：判断某个功能该不该做、先做哪版、证据强不强。
   - 新产品调研：判断一个新产品想法是否值得继续验证或启动。
   - 现状调研：梳理当前业务、流程、系统和组织问题，为改造前摸底。
2. 明确研究边界。
   - 目标对象、地区、时间窗、产品边界、组织范围、约束条件。
3. 建立证据池。
   - 二手资料：官网、价格页、财报、政策、报告、应用商店、搜索趋势、媒体与社区内容。
   - 一手资料：访谈、工单、日志、表单、报表、销售反馈、客服记录、现场观察、试投放。
4. 按类型走分支。
   - 需要市场规模、竞品格局、渠道和价格时，读 [references/research-method-routing.md](references/research-method-routing.md) 的“市场调研”部分。
   - 需要需求证据、竞品功能矩阵、优先级判断时，读 [references/research-method-routing.md](references/research-method-routing.md) 的“功能调研”部分。
   - 需要机会假设、概念测试、MVP 取舍时，读 [references/research-method-routing.md](references/research-method-routing.md) 的“新产品调研”部分。
   - 需要流程、角色、根因和例外流程诊断时，读 [references/research-method-routing.md](references/research-method-routing.md) 的“现状调研”部分。
5. 输出结论。
   - 不是只总结资料，而是输出结论、证据强度、主要风险和下一步行动。

## Core Rules

- 不把“收集资料”误当成“完成调研”。
- 至少做一次交叉验证，不用单一来源下关键结论。
- 区分“用户说想要”“用户真的受损”“用户愿意付出”。
- 区分“当前现状”“未来方案”，不要混写。
- 证据弱时明确标注“待验证”，不要硬给确定结论。

## Deliverables

- 一页版调研结论
- 证据清单
- 竞品或流程矩阵
- 风险与假设清单
- go、no-go、继续验证建议
- 下一步研究或实验计划
