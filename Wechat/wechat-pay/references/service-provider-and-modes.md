# 普通商户、服务商与接入模式

## 先做模式分流

- 如果用户问题里出现 `服务商`、`合作伙伴`、`子商户`、`二级商户`、`sp_mchid`、`sub_mchid`、`sub_appid`，优先切到 partner 文档树。
- 如果用户问题只出现商户号、公众号、小程序、App 自有接入，且没有服务商字段，优先从 merchant 文档树回答。
- 文档路径本身就是第一层分流：
  - 普通商户：`/doc/v3/merchant/...`
  - 服务商/合作伙伴：`/doc/v3/partner/...`

## 已核验的官方入口

- `接入模式`：https://pay.weixin.qq.com/doc/v3/merchant/4012068443
- `普通商户 / 开发必要参数说明`：https://pay.weixin.qq.com/doc/v3/merchant/4013070756
- `合作伙伴 / JSAPI支付 产品介绍`：https://pay.weixin.qq.com/doc/v3/partner/4012069852
- `合作伙伴 / 开发必要参数说明`：https://pay.weixin.qq.com/doc/v3/partner/4013080340

## 普通商户回答路径

1. 先看 `接入模式`
2. 再根据终端场景进入 `JSAPI` / `小程序` / `APP` / `H5` / `Native`
3. 遇到参数材料时，回到 `普通商户 / 开发必要参数说明`

普通商户常见材料通常包括：

- `mchid`
- `appid`
- 商户 API 证书
- APIv3 密钥
- 微信支付公钥 / 公钥 ID
- 平台证书

## 服务商回答路径

1. 先确认用户要查的是哪类产品；当前至少要切到 partner 文档树
2. 先开 `合作伙伴 / JSAPI支付 产品介绍` 之类的 partner 产品入口
3. 需要字段与材料时，回到 `合作伙伴 / 开发必要参数说明`

服务商模式下，回答时尤其要避免把普通商户字段直接套到服务商模式：

- 不要把 `mchid` 直接写成 `sp_mchid` 或反过来
- 不要把普通商户的 `appid` 直接当成子商户/子应用参数
- 不要把 `/merchant/` 路径下的参数总表直接给 partner 场景

## 需要保守处理的地方

- 如果用户只说“微信支付接入”，没有说明普通商户还是服务商，不要直接假设。
- 如果用户同时提到多个主体字段，先说明需要确认接入模式，再给文档路线。
- 如果需要精确参数，以 `开发必要参数说明` 和对应产品页为准，不要凭记忆列字段。
